# Token economy program — roadmap

**Дата:** 2026-05-23
**Статус:** Approved (см. [ADR-0003](../../adr/0003-token-economy-principles.md))
**Владелец:** brain_matrica (cross-project)

Этот документ — **поэтапный roadmap** реализации token economy principles. Не сам ADR (это [`adr/0003`](../../adr/0003-token-economy-principles.md)), не реестр идей. Здесь — что делаем, в каком порядке, что критерий завершения каждого этапа.

## Принцип последовательности

«Сначала видимость, потом действия» — без данных оптимизация — догадки (см. ADR-0003 принцип #1). Поэтому слой E (мониторинг) первый. Дальше — параллельно слой A (#004 minimalist docs) и слой F (CODEBASE_MAP pilot), оба опираются на замеры из E.

## Этапы

### Этап 1 — Mass-monitoring setup [STATUS: NOT STARTED]

**Цель:** иметь рабочий способ видеть «сколько тратим» в каждом проекте за последние N дней.

**Шаги:**

1. **Исследование `ccusage`** (open-source CLI для трекинга расходов Claude Code из локальных логов):
   - Запустить в brain_matrica (`npx ccusage@latest`), посмотреть формат данных
   - Понять что показывает: per-session, per-day, per-project; что не показывает
   - Альтернативы: `/cost` (built-in, текущая сессия), Anthropic Console (агрегированно, лагает)
   - Записать в `tech-radar/tools/ccusage.md` со статусом (assess/trial/adopt)

2. **Brain skill `/audit-usage`**:
   - Read `~/.claude/projects/*/` logs где Claude Code хранит сессии
   - Агрегировать: per-project, per-week, top-сессии по cost
   - Output: короткий отчёт в чат + опционально файл `docs/AI_USAGE_DASHBOARD.md` в brain
   - Запускать вручную раз в неделю; позже — `/loop 7d /audit-usage` или scheduled task

3. **Per-project практика:**
   - В CLAUDE.md каждого проекта добавить раздел «Token economy»: ссылка на ADR-0003 + правило «раз в неделю запускать `/cost` в конце сессии и фиксировать»
   - Не директива пока — рекомендация, после замеров эффекта в одном проекте — оформить через mailbox

**Критерий завершения:** есть `/audit-usage` skill в brain, есть исторические данные за минимум 2 недели хотя бы по одному проекту.

### Этап 2 — Pool #004 → директивы по проектам [STATUS: NOT STARTED, blocked by 1]

**Цель:** Pool idea [#004 minimalist AI docs](../../cross-project-ideas/ideas/004-minimalist-ai-docs-2026.md) — упразднение `DEV_HISTORY`/`DEVELOPMENT_LOG`. Перевести из «⚠️ кандидат во всех 4» в «✅ применено» через mailbox-директивы.

**Зависимость:** этап 1 — нужны метрики до/после, чтобы доказать эффект (принцип #6 ADR-0003: измеримость).

**Шаги:**

1. Замерить **до** в пилотном проекте (GONBA, в `between threads`): средний размер контекста на старте сессии, время до первой правки
2. Написать директиву `mailbox-from-brain` для GONBA: упразднить DEV_HISTORY, заменить на куратируемый snapshot
3. Подождать ack от GONBA, замерить **после** через 2-3 сессии
4. Если эффект подтверждён — серия директив по MatricaRMZ, setka, KARMAN
5. Обновить pool #004: проекты → ✅ + adaptation notes
6. Если эффект слабый — обновить ADR-0003 changelog с уроком

**Критерий завершения:** pool #004 во всех 4 проектах либо ✅, либо ⛔ (с явной причиной).

### Этап 3 — Pilot `docs/CODEBASE_MAP.md` [STATUS: NOT STARTED]

**Цель:** проверить идею «карта прежде разведки» (принцип #4 ADR-0003) на одном проекте.

**Шаги:**

1. Выбор пилота: GONBA (между нитками, средняя сложность codebase) или KARMAN (мало кода, легко начать)
2. Согласовать формат `docs/CODEBASE_MAP.md`:
   - Модули → файлы → краткое назначение (1 строка)
   - Owner / контактная зона
   - «Когда сюда лезть» — типовые триггеры
   - **Куратируемый**, обновляется при значимых архитектурных изменениях, не автогенерируется
3. Создать в пилотном проекте через директиву от brain
4. Замерить через `ccusage`: средний cost сессии до/после; число Glob/Read calls в первые 5 минут сессии
5. Если работает — оформить как новую pool-идею; перенос в другие проекты по той же схеме что #003

**Критерий завершения:** pool-идея «CODEBASE_MAP» оформлена и применена минимум в 2 проектах.

### Этап 4 — Quarterly review token economy [STATUS: RECURRING from 2026-08]

**Цель:** не дать программе превратиться в формальность.

**Что делать каждый квартал в brain_matrica-сессии:**

1. Запустить `/audit-usage` — сравнить с прошлым кварталом
2. Пересмотреть ADR-0003: какие принципы реально работают, какие добавить
3. Пересмотреть `tech-radar/tools/` — есть ли новые tools (например, обновлённый ccusage, новые Claude Code features)
4. Опросить (себя или пользователя): где Claude всё ещё «жирно» работает; что хочется автоматизировать

**Триггер:** триггерится `/start` в первой сессии нового квартала через напоминание в `SESSION_HANDOFF.md`.

## Затронутые слои (карта)

| Слой | Этап | Артефакт |
|---|---|---|
| E — Мониторинг | 1 | `ccusage`, brain skill `/audit-usage` |
| A — Контекст | 2 | директивы по pool #004 |
| F — Дорожная карта | 3 | `docs/CODEBASE_MAP.md` в проектах |
| B, C, D, G, H | по мере появления потребности | формируются на основе данных этапа 1 |

Слои B/C/D/G/H — **не запланированы конкретными этапами**, формируются по результатам мониторинга. Не строим инструменты «на всякий случай».

## Что НЕ входит в эту программу

- **Жёсткий downgrade моделей по правилу** — нарушает принцип #2. Выбор модели — за пользователем / контекстом задачи.
- **Token budget per skill с hard-stop** — то же.
- **Автогенерация документации** (CLAUDE.md, CODEBASE_MAP.md) — нарушает «куратируемое vs накопительное» (принцип #3).
- **Замена brain_matrica meta-сессий** на автоматический мониторинг — мониторинг кормит сессии данными, не заменяет их.
