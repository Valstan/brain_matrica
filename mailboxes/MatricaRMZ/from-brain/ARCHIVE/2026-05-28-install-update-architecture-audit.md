# Директива: профессиональный аудит установки/обновления клиента + дорожная карта модульности

## Контекст / мотивация

Пользователь поставил стратегическую задачу. Цель — **устойчивость прода и дешёвые обновления**: чтобы при обновлении не ломался весь клиент целиком, а сломаться могла лишь часть (которую чинят оперативно, пока остальное работает), и чтобы клиент **скачивал меньше** (не тащил весь пакет, включая неизменившийся Electron, на каждое обновление).

Конкретные подозрения пользователя:
1. При установке/обновлении на компе клиента **накапливается мусор** или файлы раскладываются неправильно.
2. Каждое обновление качает **всё целиком**, включая Electron, хотя менялись только наши модули.
3. Возможно используется **устаревшая версия Electron** — пора обновить движок.
4. Идея: разбить клиент на **отдельно обновляемые модули** и качать только затронутые. Встречный риск (его же): забывание связей между модулями → накопление ошибок и костылей. Контр-идея: «центр», который держит **дорожную карту связей** между модулями в актуальном состоянии.

Brain — meta-уровень, в код не лезу. Ниже — рамка аудита + современные мировые практики (2026), чтобы сессия MatricaRMZ стартовала с сильного брифа. **Сначала аудит и ADR с рекомендацией, реализация — отдельными релизами после.** Не сорвать текущий deep flow по BOM (v1.22.0) — взять в окно между блоками.

## Задания (разбито по блокам)

### Блок 1 — Аудит файловой раскладки установленного клиента

На реальной Windows-машине (после чистой установки и после обновления) задокументировать:
- Куда ложатся файлы: `%LOCALAPPDATA%\Programs\<app>`, `%APPDATA%\<app>` (user data, SQLite, кэш), `%LOCALAPPDATA%\<app>-updater` / pending-update кэш.
- Что **остаётся мусором**: старые версии-папки, недочищенный pending-installer, дубли `app-X.Y.Z`, осиротевший кэш блок-карт, логи без ротации.
- Чистится ли всё на обновлении и на uninstall.
- **Deliverable:** карта раскладки + список «мусора» + где именно течёт.

### Блок 2 — Аудит текущего механизма обновления

- Тип инсталлятора: NSIS (one-click?) / Squirrel.Windows / portable. (По реестру у нас GitHub Actions собирает `.exe` + `latest.yml` + torrent — разобрать что внутри.)
- Конфиг `electron-updater` / `electron-builder`: включён ли `differentialPackage` / `nsis-web`, генерится ли `.blockmap`.
- **Текущий размер закачки на одно обновление** (фактический, померить) — full installer каждый раз?
- Как раздаются апдейты: `/opt/matricarmz/updates/` + torrent + `latest.yml`. Поддерживает ли канал дифф-загрузку.
- **Deliverable:** текущий размер/обновление + ответ «дифференциальные апдейты уже работают или нет».

### Блок 3 — Дифференциальные (delta) обновления — это решает п.2 без рискованного дробления

**Главная находка для пользователя:** «качать меньше» и «не пересылать Electron каждый раз» решаются **дифференциальными обновлениями**, а НЕ разбиением клиента на модули.

`electron-builder` умеет block-map differential download (NSIS / nsis-web / generic provider): клиент качает только **изменившиеся блоки** относительно установленной версии. Неизменившиеся бинарники Electron/Chromium **автоматически не скачиваются** — block-map видит, что блоки совпадают. То есть сценарий пользователя «обновить только наши файлы, Electron не трогать» получается **из коробки**, атомарно и с rollback, без ручного дробления и без риска рассинхрона.

- Оценить включение `differentialPackage: true` (или переход на `nsis-web` target).
- Прикинуть ожидаемую экономию: типично обновление с few-MB изменений в JS/asar качает единицы МБ вместо 80-150 МБ полного инсталлятора.
- Проверить что `latest.yml` + `.blockmap` публикуются рядом с инсталлятором (и через torrent-канал тоже).
- **Deliverable:** «было vs станет» по размеру + план включения delta.

### Блок 4 — Версия Electron и cadence

- Зафиксировать **текущий major Electron** vs последний stable. Electron релизит ~каждые 8 недель и поддерживает последние **3 major**; выпасть из окна = открытые Chromium CVE (это security-долг, не только «новые фичи»).
- **Важно про идею «обновлять Electron отдельно от приложения»:** полностью отдельный авто-обновляемый рантайм Electron — **антипаттерн**. Версия Electron жёстко связана с ABI нативных модулей (`better-sqlite3` и т.п.) и с V8/Node — приложение, собранное под Electron N, не запустится на произвольной другой версии. Правильная развязка — это **тот же delta-механизм** (Блок 3): когда Electron не менялся между релизами, его блоки не качаются; когда менялся — качаются один раз вместе с релизом, собранным под него. Не нужен отдельный «канал Electron».
- **Deliverable:** разрыв версий + план апгрейда (с проверкой rebuild нативных модулей под новый ABI).

### Блок 5 — Модульность сборки и «центр связей» (ключевой вопрос пользователя)

У нас уже **pnpm-workspaces монорепо** (`electron-app/`, `backend-api/`, `shared/`, `web-admin/`, `scripts/`, `ledger/`). Это и есть фундамент. «Центр, который держит дорожную карту связей» в 2026 — это **task-graph тулинг поверх workspaces**: **Nx** или **Turborepo**.

- Граф зависимостей **выводится из импортов автоматически** (не ведётся руками) — это и есть «дорожная карта соответствий», всегда актуальная.
- `nx affected` / Turborepo affected: **собирать/тестировать только затронутые** модули + их зависимые (downstream), остальное — из кэша. Это и есть «обновлять только то, что задело».
- Защита от дрейфа и костылей (страх пользователя): (а) типизированные контракты в `shared/` как единственный источник интерфейсов между модулями; (б) integration/contract-тесты, гейтящиеся в CI по affected-графу; (в) single-version policy зависимостей. Тогда «забыть связь» нельзя — граф + типы + тесты ловят это в CI.
- **Deliverable:** оценка «Nx vs Turborepo для нашего монорепо» + что даст affected-CI (время сборки, что перестанет пересобираться).

### Блок 6 — Предостережение: НЕ дробить клиент на runtime-обновляемые модули

Прямой ответ на развилку пользователя «дробить или нет»:

- **Дробление _сборки_ (Блок 5, affected-build) — да.** Безопасно, тулинг держит связи.
- **Дробление поставляемого клиента на независимо-обновляемые runtime-модули (module federation / micro-frontends на десктопе) — НЕ рекомендуется.** Для десктопного ERP это именно тот путь, что рождает version-skew, рассинхрон IPC/контрактов и костыли ремонта, которых пользователь боится.
- Цель «ломается только часть, чиним пока остальное работает» для **клиента** лучше достигается так: клиент остаётся **единым атомарным артефактом** (но маленьким за счёт delta из Блока 3) + надёжный **rollback** на предыдущую версию при сбое апдейта.
- А **устойчивость прода** — это уже про backend: у нас он **уже dual-instance** (`primary :3001` + `secondary :3002`). Вот где живёт «часть работает, часть чиним». Усилить можно staged rollout / health-gate между инстансами — а не дроблением Electron-клиента.

## Что НЕ делать (антипаттерны)

- ❌ Не вводить отдельный авто-обновляемый «канал Electron» в обход релиза приложения — ABI-связь сломает нативные модули.
- ❌ Не дробить поставляемый клиент на независимо-версионируемые runtime-модули ради «обновлять по кусочкам» — рассинхрон контрактов дороже выигрыша.
- ❌ Не вести «дорожную карту связей» руками — пусть её выводит Nx/Turborepo из графа импортов.
- ❌ Не срывать deep flow по BOM (v1.22.0). Это аудит «в окно», результат — ADR + поэтапный план, не немедленный рефакторинг.
