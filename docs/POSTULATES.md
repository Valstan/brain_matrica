# Постулаты brain_matrica

Книга накопленных принципов работы. **Куратируемый список**, не автогенерируется. Каждый постулат — короткое утверждение + одна строка «почему». Если постулат устарел — не удалять, а помечать `~~зачёркнутый~~` + ссылка на superseding ADR.

Открыт для пополнения от сессии к сессии — это «память на будущее».

## I. Архитектура brain ↔ projects

1. **brain читает проекты только в read-only режиме.** Никакого редактирования файлов чужих проектов из сессии brain_matrica. Исключение — `projects/<P>.md` (это файл реестра brain'а). _Почему:_ роль brain — куратор-хаб, не разработчик; конфликты с проектными сессиями исключаются._ [ADR-0001](../adr/0001-brain-projects-mailboxes.md)._
2. **Вся коммуникация brain ↔ projects идёт через mailboxes — асимметрично (v3).** brain пишет/коммитит только в свой репо (`brain_matrica/mailboxes/<P>/from-brain/`). Проект пишет/коммитит только в свой репо (`<project>/mailbox/to-brain/`). Каждая сторона **читает** другую через `git pull --ff-only` (read-only). _Почему:_ кросс-репо коммиты приводили к конфликтам, дублям и размытой ответственности; асимметрия их исключает._ [ADR-0001 v3](../adr/0001-brain-projects-mailboxes.md#asymmetric-scheme)._
3. **Peer-to-peer писем между проектами нет.** Если проект А хочет передать что-то проекту B — пишет в свой `mailbox/to-brain/`, brain читает и форвардит созданием нового письма в `mailboxes/B/from-brain/`. _Почему:_ сохраняет hub-роль brain'а; идеи не расходятся без курации._
4. **Проект проверяет почту в каждом `/start`.** При наличии писем — доклад до основного workflow. `high-urgency` поднимается даже если письмо одно. _Почему:_ нет риска прозевать важное._
5. **ARCHIVE никогда не удаляется.** Архивные письма — это история обмена. Чистка только осознанным архивированием старых писем в `_old/`, не покомитному удалению. _Почему:_ восстанавливаемость контекста; git blame теряет данные при удалении._
6. **Каждое письмо kind=idea/directive имеет `compliance`-уровень.** Три значения: `suggest` (MAY — рассмотри, можно проигнорировать), `recommend` (SHOULD — применить с адаптацией, отказ обосновать), `mandate` (MUST — безусловное выполнение, отказ только при техническом блокере). _Почему:_ kind отвечает на «что это», urgency — «когда читать», compliance — «насколько обязательно»; без этой оси brain не может различать «попробуй» и «сделай»._ [ADR-0001 §Compliance levels](../adr/0001-brain-projects-mailboxes.md#compliance-levels)._

## II. Стиль документов

6. **Один файл — одна мысль.** Идея / решение / постулат / письмо — отдельный markdown. _Почему:_ удобство ссылок, изоляция изменений, нет монолитов которые никто не читает._
7. **INDEX.md — навигация, не содержимое.** В индексных файлах только короткая таблица со ссылками. _Почему:_ страницы навигации не должны соревноваться с тем, на что они указывают._
8. **Краткость важнее красоты.** Списки и таблицы — предпочтительнее длинных параграфов. _Почему:_ в meta-репо ищут конкретный факт, не читают подряд._
9. **Финальные сообщения пользователю — на русском, идентификаторы и commit messages — на английском.** _Почему:_ user-facing readability + portable git history._

## III. Управление проектами

10. **Реестр проектов в `projects/INDEX.md` — куратируемый, не автогенерируемый.** brain не пытается строить список по содержимому `WORKSPACE_ROOT/` автоматически. _Почему:_ не каждый клон на компе — это управляемый проект; включение явное._
11. **Kill-policy — триггер для разговора, не автоматическое действие.** Условия из [projects/INDEX.md](../projects/INDEX.md) → brain поднимает вопрос; решение всегда за пользователем. _Почему:_ автоархивация проекта без обсуждения — потеря работы._
12. **Все клоны проектов лежат в одной `WORKSPACE_ROOT`.** brain_matrica — тоже клон в этой папке. Относительные пути `../<project>/` всегда работают. _Почему:_ переносимость между компами; bootstrap нового компа сводится к клонированию папки._ См. [BOOTSTRAP](BOOTSTRAP.md)._
13. **Имя локальной папки = имя GitHub-репо (case-sensitive).** Никаких aliases типа `gonba`-папка-для-`Gonba`-репо. _Почему:_ убирает класс ошибок с case-mismatch на cross-platform._

## IV. Память и накопление знаний

14. **Знания делятся на личную память (моя) и репо-память (видна пользователю).** Личная — в `~/.claude/projects/<slug>/memory/`. Репо-память — этот файл + ADR + cross-project-ideas. _Почему:_ часть знаний имеет смысл только для AI-агента (контекст, предпочтения), часть — общая для всех сессий._
15. **Постулат добавляется когда правило применимо в N+1 ситуации.** Однократное решение — это просто решение, не постулат. _Почему:_ список постулатов не должен раздуваться от каждой реплики._
16. **Устаревший постулат не удаляется.** Только зачёркивается + ссылка на superseding. _Почему:_ история мышления brain'а — это тоже знание; для будущей сессии полезно понять «почему мы отказались от X»._

## VI. Git и flow

17. **Никаких прямых push в `main` / `master` / default branch.** Любые изменения через `feat/`-`fix/`-`chore/` ветку + PR + merge. _Почему:_ audit trail, atomic revert через `git revert <merge-commit>`, защита от случайных пушей AI-агентов; auto-mode classifier Claude Code блокирует direct push на main, и мы превращаем баг в feature. _Исключение:_ hot-fix аварии прода (см. §8 [ADR-0002](../adr/0002-pr-only-flow-no-direct-push.md))._
18. **Имя ветки — feature-префикс + slug:** `feat/<slug>`, `fix/<slug>`, `chore/<slug>`, `docs/<slug>`, `refactor/<slug>`. _Почему:_ читаемая история веток, фильтрация по типу._
19. **PR создаётся через `gh pr create`** с Summary + Test plan в body. Title под 70 символов. _Почему:_ структурированный audit trail; CLI быстрее веб-интерфейса._
20. **Ветка удаляется после merge** (`gh pr merge --delete-branch`). _Почему:_ список веток на GitHub не должен превращаться в кладбище._
21. **Force-push в feature-ветку разрешён**, force-push в main — **запрещён**. _Почему:_ feature-ветка — приватная для разработчика, можно rebase / amend; main — общая, force-push на ней теряет историю._

## V. Когда писать сюда

- Согласовали с пользователем явное правило, которое будет применяться **во всех будущих сессиях** brain_matrica
- Приняли cross-project ADR — добавляем короткий постулат отсюда со ссылкой на полный ADR
- Заметили закономерность работы (формат коммитов, стиль ответов, привычка) — пользователь подтвердил, что это правило, не предпочтение момента

Когда **не** писать:
- Tactical детали одного проекта — это в `projects/<P>.md`
- Сиюминутные задачи — это в `mailboxes/<P>/from-brain/` (директивы проектам) или `cross-project-ideas/`
- Личные предпочтения пользователя без обобщённого правила — это в личной памяти агента
