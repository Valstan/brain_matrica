# Директива: финальный аудит сквозной логики перед запуском в эксплуатацию

## Контекст

MatricaRMZ приближается к запуску в реальную эксплуатацию ремонтного завода двигателей. Нужна финальная end-to-end проверка что **вся сквозная цепочка работает целостно** — не отдельные модули, а связки между ними.

Это не точечный bug-fix и не feature. Это **acceptance audit** — последний просмотр глазами «инспектора» прежде чем настоящие двигатели и настоящие заказчики начнут попадать в систему.

## Цель аудита

Прогнать всю логическую цепочку жизни двигателя в системе **end-to-end** и ответить на 7 вопросов:

1. Все ли этапы реализованы и работают без багов?
2. Есть ли недоделанные места которые в реальной работе создадут проблемы?
3. Есть ли точки улучшения которые сделают учёт проще и удобнее — **без добавления бюрократии**?
4. Все ли взаимосвязи между этапами работают корректно?
5. Все ли важные данные попадают в учёт — нет ли «теневых» мест где что-то теряется?
6. Видит ли отчёт-прогноза все данные которые должен видеть?
7. Спецификации сборки (BOM) обрабатываются ли правильно с учётом текущего BOM-refactor?

## Цепочка для проверки (по стадиям жизни двигателя)

Пользователь перечислил стадии так — **сверь с реальной терминологией в коде MatricaRMZ** и адаптируй под фактический набор стадий:

1. **Заключение контракта** с заказчиком — создание договора, его поля, связь с заказчиком
2. **Приёмка двигателя на завод** — регистрация поступления, идентификация, начальная карточка
3. **Дефектация** — фиксация состояния, перечень дефектов, фотофиксация если есть
4. **Разборка** — учёт извлечённых деталей, их состояние, какие подлежат восстановлению / замене
5. **Ремонт** — восстановление деталей, закупка новых, учёт работ и материалов
6. **Сборка** — по BOM-спецификации, фиксация «какая деталь куда пошла»
7. **(возможно)** Испытания — приёмочные тесты после сборки
8. **(возможно)** Отгрузка / выдача заказчику — закрытие договора

**Если в системе есть стадии которых нет в этом списке** (например, технологическая подготовка между дефектацией и разборкой, или согласование сметы между дефектацией и началом ремонта) — обязательно включи их в аудит. Пользователь говорил «и какие там еще есть и какие были задуманы в проге» — то есть он сам хочет увидеть полный список.

## Что проверять на каждой стадии

### A. Создание сущностей
- При создании новой записи на стадии — **все ли обязательные поля** заполняются?
- Есть ли поля которые «вроде есть в форме» но фактически нигде не используются (мусор — кандидат на удаление)?
- Есть ли поля которые **должны быть** но их нет (например, фотография дефекта, ФИО мастера на разборке)?

### B. Связи между сущностями
- Запись на стадии N — корректно ли ссылается на запись на стадии N-1?
- Можно ли через UI восстановить полную цепочку: «двигатель X → дефектация → разборка → деталь Y → ремонт Z → сборка W»?
- Есть ли cascade-удаление / soft-delete где ожидается, и нет где не должно быть (удалили двигатель — не должны потеряться записи о его ремонте в архиве)?

### C. Учёт (нет ли «теневых» мест)
- Все ли точки где деталь / материал / работа меняет состояние — фиксируются в БД?
- Можно ли потом из отчётов восстановить хронологию: «кто, когда, что сделал»?
- Не утекают ли данные в каком-то месте мимо учёта (например, ремонт списал материалы, но в отчёт по материалам не попало)?

### D. Reports / прогнозы
- Каждый отчёт — видит ли он **все** релевантные данные?
- Отчёт прогноза по загрузке — учитывает ли он все стадии, или только некоторые?
- Если в системе есть концепция «двигатель сейчас в стадии X» — есть ли отчёт «сколько двигателей в каждой стадии» и работает ли он?
- Есть ли стадии которые «невидимы» для отчётов (например, двигатель «между стадиями» нигде не показан)?

### E. BOM-спецификации (контекст текущего refactor)
- После завершения BOM-refactor 5 релизов — связан ли реальный поток сборки с BOM как было задумано?
- Если сборщик отметил «деталь X использована» — обновляется ли BOM-учёт автоматически?
- Если в BOM указана деталь A, а фактически использовали аналог B — как это фиксируется (free-text? отдельное поле «фактическая замена»? никак)?

## Подход к проведению

1. **Тестовый прогон сценария** — заведи в системе фейкового двигателя «End-to-end audit 2026-XX», прогони его через все стадии, проверь что каждый шаг работает и виден везде где должен.
2. **Audit чек-лист** — пройди по каждой стадии и по каждому пункту A-E из этого письма, веди заметки.
3. **Cross-check с прежними задачами** — посмотри в `docs/PENDING_FOLLOWUPS.md` нет ли уже зафиксированных проблем по этой теме (могут быть «старые» баги стадий которые ждут решения).

## Что НЕ делать

- ❌ **Не превращать в бюрократию.** Пользователь явно сказал: «удобнее и проще, но при этом не погрязнуть в бюрократии». Если фича работает «нормально для рабочего», не добавляй обязательных полей ради добавления.
- ❌ **Не переписывать архитектуру по ходу аудита.** Если найдёшь большую архитектурную проблему — зафиксируй её отдельной задачей (`docs/PENDING_FOLLOWUPS.md` + при необходимости ADR), не лезь чинить в этом же подходе.
- ❌ **Не делать аудит во время активного релизного цикла BOM-refactor.** Тестировать движущуюся цель бессмысленно — после очередного релиза часть выводов аудита устареет. Дождись завершения текущей нитки (v1.21.7 или сколько там запланировано в `docs/plans/bom-refactor-2026-05.md`) и тогда начинай аудит уже стабильной системы.
- ❌ **Не путать аудит с «всё переделать».** Цель — найти что не работает / не достаёт / можно улучшить точечно. Не «всё снести и сделать заново».
