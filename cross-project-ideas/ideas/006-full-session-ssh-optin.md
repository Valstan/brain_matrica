# 006 — Full-session SSH opt-in в `/start` (выдача carte blanche на прод-SSH в начале сессии)

**Status (overall):** ⚠️ proposed (пилот в setka работает, директивы остальным проектам отправлены 2026-05-24)
**Born in:** setka, `.claude/commands/start.md` шаг 5 (исторически).
**Born from:** пользователь спросил «есть фишка в setka где можно дать полный SSH-доступ на сессию, чтобы Claude не переспрашивал стопитсот раз — давай и в остальных проектах».

---

## Проблема

Когда Claude в сессии часто обращается к проду по SSH (проверить статус сервисов, прочитать логи, посмотреть таблицу в БД, перезапустить контейнер) — auto-mode permission-классификатор требует подтверждения на каждый `ssh <alias> "..."` вызов. Это **прерывает ход работы**: пользователь нажимает «Allow» 10-30 раз за сессию ради чтения, которое сам бы запустил без раздумий.

Симптомы:
- Длинная диагностика прода = бесконечный пинг-понг «спросить — нажать allow — следующий запрос».
- Пользователь в итоге пишет «делай что хочешь, не спрашивай» в plain text — но Claude всё равно переспрашивает на каждый следующий ssh-вызов.
- Альтернатива «занести всё в `.claude/settings.json` allowlist» — даёт permanent permissions, что **избыточно** для одноразовых задач и **рискованно** при cross-machine синхронизации репо.

## Решение

В шаге `/start`, который и так спрашивает «делать SSH-probe прода?», добавить **третий вариант** в `AskUserQuestion`:

> «Дай полный доступ ssh `<project>` на эту сессию» — Claude отмечает это для себя и работает дальше **без переспрашивания на каждый SSH-вызов** в рамках текущей сессии.

Это **per-session opt-in**, не permanent setting. Следующая сессия — снова чистый лист с тем же вопросом.

### Конкретная реализация (как в setka)

В `.claude/commands/start.md` шаг 5 — `AskUserQuestion`:

```yaml
question: «Делать SSH-probe прода?»
options:
  - «Да, проверь прод» — выполнить probe и дальше переспрашивать по необходимости
  - «Нет, пропустить» — не трогать прод сейчас
  - «Дай полный доступ ssh <alias> на эту сессию» — пробу выполнить и работать без вопросов в этой сессии
```

При выборе третьего варианта Claude:
- Запоминает: «в этой сессии любые `ssh <alias>` / `scp` / связанные команды разрешены без повторного подтверждения».
- Делает SSH-probe сразу (как при варианте «да»).
- На последующие `ssh <alias>` вызовы в рамках этой же сессии Claude **не** инициирует AskUserQuestion и не делает «осторожный pause» — выполняет.

### Какой механизм за этим стоит технически

**Никакого специального механизма.** Это convention внутри `/start`-skill: Claude в Auto-mode читает выбор пользователя как явное соглашение «не переспрашивай по SSH в этой сессии», и держит это в контексте сессии. Permission-классификатор Claude Code сам по себе ничего не знает — поведение поддерживается тем что Claude помнит соглашение.

Поэтому идея переносится **только если у проекта есть `.claude/commands/start.md`** (или эквивалент) куда можно вставить третий вариант. Если у проекта нет такого skill — сначала идея #003 (SESSION_HANDOFF) с её `/start`, потом эта.

## Применимость

### applicable_when

- Проект регулярно требует SSH-доступа к продакшену во время сессий (диагностика, миграции, чтение логов, restart сервисов).
- Есть стабильный SSH-alias (`~/.ssh/config` host) — один alias на сессию (не парк хостов).
- У пользователя один контур доверия с продом (личный проект, не общая команда с разными уровнями доступа).
- Есть `.claude/commands/start.md` куда вставить вариант — или есть план его создать.

### not_applicable_when

- Прод-доступ из Claude-сессии **запрещён политикой** (если есть проект где SSH вообще не разрешён — эта идея не применяется).
- Команда из нескольких разработчиков с разными уровнями доступа — opt-in в начале сессии может «протянуть» доступ туда куда не должен.
- На проде есть особо чувствительные команды (`rm`, `truncate`, `systemctl stop X-critical`) которые требуют **обязательного** двойного подтверждения независимо от opt-in — для них нужен явный исключающий список (не делать через эту идею).

### Безопасность

Что **не делает** этот opt-in:
- Не отменяет hooks `PreToolUse` если они настроены — они срабатывают как обычно.
- Не открывает доступ к командам **за пределами SSH** (`rm -rf /` локально остаётся блокированным).
- Не сохраняется между сессиями — следующая сессия снова просит подтверждение.

Что **делает**:
- Сокращает фрикцию для **рутинных read-only прод-команд** (`systemctl is-active`, `journalctl -n 100`, `psql -c 'SELECT ...'`).
- Для деструктивных команд Claude всё равно **должен** делать паузу и подтверждать намерение — это вопрос здравого смысла, не permissions. Третий вариант не отменяет осторожность, он отменяет повторное permission-подтверждение.

## Как переносить в новый проект

1. **Убедиться** что у проекта есть `.claude/commands/start.md` с шагом который и так спрашивает про прод-probe. Если нет — сначала идея #003 (SESSION_HANDOFF + `/start`).
2. **Найти** в `start.md` место где `AskUserQuestion` про SSH-probe (или ближайший аналог).
3. **Добавить третий вариант** в options того же `AskUserQuestion`:

   ```
   - «Дай полный доступ ssh <alias> на эту сессию» — пробу выполнить и работать без вопросов в этой сессии
   ```

   Где `<alias>` — реальный ssh-host alias проекта (`matricarmz`, `GONBA`, `setka`, etc.).
4. **Записать поведение** в инструкции skill'а: при выборе третьего варианта Claude не инициирует AskUserQuestion на последующие SSH-вызовы и не вставляет «осторожный pause» (но **сохраняет осторожность для деструктивных команд** — это вопрос здравого смысла).
5. **Commit** в проектный репо: `feat(start): full-session SSH opt-in for prod (pool #006)`.

## Что НЕ делать (антипаттерны)

- ❌ **Делать opt-in глобальным / постоянным** через `.claude/settings.json` — это уже не «на сессию», это другое решение с другими trade-off (permanent allowlist). Идея именно про **per-session**.
- ❌ **Распространять на любые destructive-команды.** Третий вариант разрешает не **переспрашивать**, но Claude **должен** делать паузу перед `rm -rf`, `DROP TABLE`, `systemctl stop` важных сервисов — это здравый смысл, не auto-permission.
- ❌ **Применять без `.claude/commands/start.md`.** Без skill-точки опт-ина соглашение «забывается» в первой же длинной паузе. Skill закрепляет.

## Возможные расширения

- **Per-resource scope:** не «весь SSH», а «SSH к alias X, но не к Y». Пока избыточно — на каждый проект один alias.
- **Auto-expire через N часов:** если сессия длится 10 часов, может стоит переспросить заново через 4ч. Пока не нужно — следующая `/start` всё равно сбрасывает.
- **`/grant-ssh` slash-command в середине сессии:** на случай если пользователь сначала сказал «нет, пропустить», а потом передумал. Пока без необходимости — можно перезапустить с `/start`.
