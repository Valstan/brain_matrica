# ADR 0003: Token economy principles

**Date:** 2026-05-23
**Status:** Accepted
**Applies to:** brain_matrica, MatricaRMZ, GONBA, setka, KARMAN (и все будущие проекты под управлением)

## Context

Стоимость работы с LLM растёт пропорционально размеру контекста + количеству tool-вызовов + выбранной модели. Без явных принципов и инструментов мониторинга проекты под управлением brain_matrica могут тратить токены избыточно: повторное чтение тех же файлов, недостаточно сфокусированные skills, использование Opus там где хватает Sonnet, разведка codebase «на ощупь» при каждом старте сессии.

При этом цель — **не сэкономить любой ценой**, а сэкономить **без потери качества и скорости разработки**. Слишком жёсткая экономия (запрет читать достаточно контекста, downgrade модели на сложных задачах) ведёт к ошибкам, переделкам, лишним итерациям — итог дороже исходной «экономии».

## Decision

Принимаем cross-project дисциплину «token economy» из шести принципов:

### 1. Видимость прежде оптимизации

Без данных о расходах оптимизация — догадки. Каждый проект должен иметь способ увидеть «сколько тратим сейчас» (минимум: `/cost` per session, по возможности — агрегированный `ccusage` или эквивалент). Раз в неделю — короткий audit в brain_matrica-сессии.

### 2. Качество и скорость не приносим в жертву

Если экономия токенов делает Claude:
- Вынужденным переспрашивать пользователя там где раньше не переспрашивал
- Совершающим больше ошибок (нужно больше итераций fix-fix-fix)
- Медленнее в реальной разработке

— это **не экономия**, это сдвиг расходов. Откатить.

### 3. Контекст — куратируемый, не накопительный

`CLAUDE.md` и связанные источники правды должны грузиться **избирательно**. Растущие append-only журналы (`DEVELOPMENT_LOG`, `DEV_HISTORY`) — антипаттерн эры слабых LLM, см. pool [#004](../cross-project-ideas/ideas/004-minimalist-ai-docs-2026.md). Замена — короткий куратируемый snapshot + `git log` для истории + `SESSION_HANDOFF.md` для нитки.

### 4. Карта прежде разведки

В каждом проекте — `docs/CODEBASE_MAP.md` (или эквивалент): где живёт что, owner, «когда сюда лезть». Это снижает расходы на discovery-фазу (Glob/Read «на ощупь») и устраняет необходимость в длинных `AI_DEV_GUIDE`. Карта куратируется человеком + Claude по согласию, не автогенерируется.

### 5. Подходящий инструмент — подходящей задаче

- Параллельные tool calls когда нет dependency
- Sub-agents для разведки (изолируют контекст)
- Routing моделей: простое — Haiku/Sonnet/`/fast`; архитектурное — Opus
- `/compact` после длинных сессий вместо новой сессии
- Prompt caching: стабильный контекст в начале (CLAUDE.md, источники правды) — не менять без нужды в течение сессии

### 6. Экономия — измеримая

Каждое нововведение из этой программы должно **до/после** показать дельту: либо в `ccusage`-данных, либо в «время сессии», либо в «число tool calls на сравнимой задаче». Без замера — нововведение не считается принятым.

## Consequences

**Положительные:**
- Долгосрочное снижение себестоимости разработки во всех 4 проектах
- Системный мониторинг — ранее замечаемые антипаттерны (большие чтения, повторы)
- Дисциплина документации — меньше «мусора» который Claude обязан читать

**Отрицательные / стоимость:**
- Время на внедрение мониторинга (ccusage + per-project практика)
- Куратирование `CODEBASE_MAP.md` — постоянная активность
- Риск перегиба (слишком короткие промты → больше итераций) — митигируется принципом #2

## Alternatives considered

- **A. Ничего не делать, оптимизация естественным образом случится.** Отвергнуто: без видимости (принцип 1) нет обратной связи; антипаттерны накапливаются.
- **B. Жёсткие лимиты — token budget per skill, обязательный downgrade модели.** Отвергнуто: ломает принцип 2; ущерб качеству.
- **C. Только мониторинг, без принципов и пилотов.** Отвергнуто: данные без действий — отчёт ради отчёта. Этот ADR фиксирует и принципы (что считаем правильным), и обязательство action (план реализации).
