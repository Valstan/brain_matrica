# 005 — `docs/CODEBASE_MAP.md` — куратируемая карта монорепо для узкого cold-start

**Status (overall):** ⚠️ proposed (pilot MatricaRMZ, замер эффекта 2026-06-07)
**Born in:** brain_matrica, 2026-05-24 (этап 3 token economy program)
**Born from:** [ADR-0003 принцип #4](../../adr/0003-token-economy-principles.md) «Карта прежде разведки» + анализ baseline (MatricaRMZ — 66% всех расходов, top-сессии $113 из-за энциклопедических cold-start'ов).

---

## Проблема

Cold-start сессии в большом монорепо стоит дорого:

- `/start` без явной задачи разворачивается в «прочитай весь `docs/` для понимания общей картины» — десятки файлов на первичное чтение, prompt cache не помогает (TTL 5 мин, между сессиями всегда холодный).
- Glob/Read «на ощупь» в первые 5-10 минут — Claude ищет где живёт фича, попутно читая половину репо.
- `AI_DEV_GUIDE` / `ARCHITECTURE.md` накопительные → длинные → читать их сами по себе дорого.
- Baseline MatricaRMZ: top-2 сессии `e18945ec` ($113.76, 8.5ч) и `6cc0759f` ($113.62, 6ч) — обе именно с длинным энциклопедическим cold-start.

Симптом — Claude в первые 5 минут делает >10 Glob/Read без явной цели, «осваивается».

## Решение

`docs/CODEBASE_MAP.md` — **куратируемый** markdown ≤2 экрана. Объясняет **где что живёт** и **когда сюда лезть**. Дополняет [SESSION_HANDOFF.md](003-session-handoff.md) (которая отвечает на «что сейчас активно»).

Три качества карты:
1. **«Где живёт X»** — навигация от понятия к файлу за один взгляд (таблица «Пакет / Модуль → Когда лезть»).
2. **«Когда сюда лезть»** — триггеры (типовые задачи, симптомы багов). Не описание кода, а инструкция «при каком запросе пользователя открыть этот модуль».
3. **«Что сейчас активно»** — нитки и горячие зоны, чтобы новая сессия сразу понимала контекст. Перекрывается с SESSION_HANDOFF, но карта даёт **где** этой нитки код, а HANDOFF — **что** делается.

### Подключение в `/start`-skill

После `SESSION_HANDOFF.md` (нитка) — `docs/CODEBASE_MAP.md` (карта). Заменяет привычку «прочитать весь `docs/` подряд». Конкретные файлы вычитываются **только когда задача того требует**.

### Куратирование

- Обновляется **при значимых изменениях** (новый модуль, переименование пакета, перенос ответственности, окончание нитки).
- **Не** обновляется при каждом commit-fix или мелком файле — карта не log.
- Обновлять может Claude в рамках сессии где изменения происходят — это часть работы, не отдельный шаг.

### Соотношение с другими файлами

| Файл | Объём | Когда читать | Что хранит |
|---|---|---|---|
| `SESSION_HANDOFF.md` | ≤1 экран | В начале каждой сессии (шаг 0) | Одна текущая нитка, следующий шаг |
| **`CODEBASE_MAP.md`** | **≤2 экрана** | **В начале каждой сессии (шаг 1)** | **Где живёт что + когда сюда лезть** |
| `DEVELOPMENT_LOG.md` / `DEV_HISTORY.md` | Растёт | По требованию (история) | История работы |
| `PROJECT_STATE.md` | Стабильный | По требованию (правила) | Архитектурные правила |
| `ARCHITECTURE.md` (если есть) | Длинный | По требованию (глубокое погружение) | Детальная архитектура |

CODEBASE_MAP — **точка входа**, не полная дока. Если хочется детальнее — `docs/architecture/<module>.md` мелкие, по требованию из карты.

## Применимость

### applicable_when

- Монорепо или большой codebase (>~30k LOC, >5 пакетов / модулей).
- Cost-heavy проект (Claude-сессии регулярно стоят >\$30 за пилот, длинные cold-start'ы заметны в `/audit-usage`).
- Уже применена идея [#003 SESSION_HANDOFF](003-session-handoff.md) — карта стыкуется с handoff'ом в `/start`-skill.
- Есть привычка `/start`-cold-start: первая Claude-сессия дня / недели часто длится >2 часов.

### not_applicable_when

- Маленький проект (<~10 файлов исходников) — карта будет дороже самого репо.
- Codebase меняется очень часто (rapid prototyping) — карта будет постоянно отставать.
- Нет per-project `.claude/commands/start.md` — карту некуда подключить, она зависнет.
- Проект уже использует подробный `ARCHITECTURE.md` который **работает** (Claude не делает энциклопедических cold-start) — нет проблемы, не нужно решение.

## Как переносить в новый проект

1. Создать `docs/CODEBASE_MAP.md` — куратируемая карта **под факт codebase** (структуру таблиц адаптируй, не копируй слепо из MatricaRMZ). Три обязательных свойства: «где живёт X», «когда сюда лезть», «что сейчас активно».
2. Обновить `CLAUDE.md` — добавить `docs/CODEBASE_MAP.md` в источники правды, явная фраза «не вычитывать `docs/` подряд — опираться на эту карту».
3. Обновить `.claude/commands/start.md` — **шаг 1** (после SESSION_HANDOFF): `docs/CODEBASE_MAP.md`. Источники правды (`DEVELOPMENT_LOG`, `PENDING_FOLLOWUPS`, `PROJECT_STATE`) — читать **только если активная нитка их требует**, не безусловно.
4. Commit + push: `docs(codebase-map): initial map for narrow cold-start (token economy etap 3)`.

## Что НЕ делать (антипаттерны)

- ❌ **Автогенерация.** Скрипт «обойди дерево, выведи список файлов» — это не карта. Карта объясняет **смысл** и **когда сюда лезть**. Автогенерация = очередной длинный текст который Claude должен будет вычитать.
- ❌ **Энциклопедия.** Карта ≤2 экрана. Если хочется детальнее — отдельный `docs/architecture/<module>.md` куда указывает карта, по требованию.
- ❌ **Подмена SESSION_HANDOFF.** Карта = «где всё», HANDOFF = «что сейчас». Разные слои, оба нужны.
- ❌ **Одновременно с упразднением DEVELOPMENT_LOG ([pool #004](004-minimalist-ai-docs-2026.md)).** Это разные изменения, риск перепутать вклады при замере. Сначала карта, потом #004.

## Замер эффекта (общая методика, не только MatricaRMZ)

После применения — 2-3 сессии в проекте работают **с** картой. brain через 2 недели в `/audit-usage`:

1. **Метрика 1:** средний cost сессии проекта (целевая дельта: -20% или больше).
2. **Метрика 2:** топ-сессии проекта (целевая дельта: было \$X → стало \$0.8X или меньше при том же объёме работы).
3. **Метрика 3 (qualitative):** делает ли Claude меньше Glob/Read «на ощупь» в первые 5 минут.

Если эффект слабый или отрицательный — карта пересматривается / упраздняется. ADR-0003 принцип #6: измеримость.

## Возможные расширения (на потом)

- **Авто-проверка stale карты:** если карта не обновлялась 60 дней при наличии нетривиальных коммитов в `src/` — флаг в `/start`-отчёте.
- **Cross-project консолидация:** если 3+ проектов внедрят карты — собрать в brain_matrica мета-наблюдения о том какая структура карты работает лучше.
