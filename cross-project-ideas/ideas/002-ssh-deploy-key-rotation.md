# Периодическая ротация SSH-deploy-ключей

## Проблема

Даже изолированный per-project ключ (см. [[001-isolated-deploy-ssh-key]]) может утечь — через GitHub secret leak, скомпрометированный CI runner, украденный ноут или просто старый бэкап. Если мы не узнаем об утечке, доступ остаётся **вечным**.

Регулярная ротация ограничивает окно эксплуатации: даже если ключ ушёл наружу, через N дней он становится бесполезным.

## Решение

1. Зафиксировать дату создания ключа в репозитории (например в `docs/PROJECT.md`).
2. Зафиксировать целевую периодичность (90/180/30 дней — зависит от уровня риска).
3. В `/start` (или эквивалентной команде открытия сессии) проверять возраст ключа и предупреждать заранее (например, за 10 дней до дедлайна).
4. Когда время пришло — повторить процедуру создания ключа из [[001-isolated-deploy-ssh-key]] с новой парой, новым именем файла (или просто перегенерировать `_deploy` ключ), обновить authorized_keys и GH secret.

## Рекомендуемые периоды

| Сценарий | Период |
|---|---|
| Публичный сайт, прод с пользовательскими данными | 90 дней |
| Внутренний сервис, ограниченный доступ | 180 дней |
| Прод с высокими security-требованиями (платежи, PII) | 30 дней |
| Stage/dev-only | 365+ дней или не ротируем |

Пользователь явно сказал — для разных проектов разные периоды. Согласуй с пользователем при первом внедрении.

## Шаги внедрения (для нового проекта)

1. В `docs/PROJECT.md` добавь блок:
   ```markdown
   ## SSH deploy-ключ — ротация

   - **Файл:** `~/.ssh/id_ed25519_<proj>_deploy`
   - **Создан:** YYYY-MM-DD
   - **Период ротации:** N дней
   - **Следующая ротация не позднее:** YYYY-MM-DD
   - **Процедура:** см. cross-project ideas pool, ideas/001-isolated-deploy-ssh-key.md
   ```
2. В slash-команде `/start` (или эквиваленте) добавь шаг: прочитать эту секцию, посчитать дельту от сегодняшней даты, если осталось < 10 дней — поставить в отчёт **🟡 напоминание о ротации**.
3. При ротации — повторить процедуру из [[001-isolated-deploy-ssh-key]], обновить даты в `docs/PROJECT.md`.

## Подводные камни

- **Запоминай в РЕПО, не в memory.** Memory persists через сессии Claude, но не доступна когда пользователь сам открывает проект через год без AI. Дата ротации = факт о проекте, должен жить в git.
- **Не ротируй посреди инцидента.** Если CI горит — сначала пофиксь, потом ротируй.
- **Чем чаще ротация — тем выше шанс ошибки.** Слишком короткий период (например 7 дней) — больше риск потери доступа, чем пользы от безопасности. 30–90 дней оптимально.
- **Старый ключ удаляй из authorized_keys, иначе ротация бессмысленна.** После проверки что новый работает.
