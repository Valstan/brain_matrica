# 009 — Рефлекс шеринга находок (условный шаг в `/close_session`)

**Status (overall):** ⚠️ in adoption (setka pioneer de-facto; MatricaRMZ/GONBA — директивы 2026-05-29; KARMAN — в связке с [#003](003-session-handoff.md))
**Born in:** brain_matrica (meta-сессия 2026-05-29, по запросу @valstan).
**Born from:** канал «проект → мозг» (`mailbox/to-brain/`, [ADR-0001](../../adr/0001-brain-projects-mailboxes.md)) существует, но за всё время проекты прислали **ровно одну** само-инициативную находку (setka → паттерн секретов, ставший pool [#008](008-secrets-outside-repo.md)). Всё остальное в `to-brain/` — ack'и и done-репорты на запросы мозга. Владелец хочет, чтобы проекты **сами** делились новыми скиллами, а не только по явной просьбе.

---

## Проблема

Шеринг находок сейчас **pull + редкий спонтанный push**:

- мозг читает `to-brain/` только на своём `/start` (pull);
- у проекта нет триггера/рефлекса, обязывающего поделиться переносимой фишкой;
- эмпирика: само-инициативное письмо случилось 1 раз за всё время существования канала.

Риск: значимый скилл/паттерн, рождённый в проекте X, не доходит ни до мозга, ни до проектов Y/Z. Владелец «может пропустить что-то новое и важное».

## Решение

**Условный рефлекс** на стороне проекта — шаг в `/close_session` (рядом с заполнением `SESSION_HANDOFF`):

> В конце сессии спроси себя: *родилась ли значимая переносимая находка?* Прогони через фильтр. Если **да** — напиши `mailbox/to-brain/YYYY-MM-DD-<slug>.md`. Если нет — **ничего не пиши**.

Плюс строка-напоминание в `CLAUDE.md` проекта (раздел про связь с brain).

**Тишина = норма** (как пустой inbox). Цель — не «писать каждую сессию», а «не пропустить значимое».

## Фильтр (анти-спам — делиться только если ВСЕ три выполнены)

1. **Значимость** — новый скилл / фича / паттерн / решённая нетривиальная боль; не рутинный фикс, не бамп зависимости, не локальный рефактор, не релиз.
2. **Переносимость** — применимо за пределами домена этого репо (другой проект мог бы переиспользовать).
3. **Неочевидность** — без этой работы сам бы не знал; есть «урок».

Любое «нет» → не пишем. **По умолчанию — не пишем.** Лучше пропустить пограничное, чем завалить мозг пустяками.

### Калибровка

- ✅ **Делиться:** «секреты вне репо» (setka→#008), «CODEBASE_MAP для cold-start» (#005), «изоляция SSH deploy-key» (#001), новый CI/деплой-паттерн, найденный обход бага инструмента, удачный приём token-economy.
- ❌ **Молчать:** «пофиксил баг в нашем `<domain>`-модуле», «обновил зависимость», «отрефакторил наш компонент», «выпустил vX.Y.Z» — специфично/рутинно.

## Формат письма (проект → brain)

Канонический шаблон — письмо-источник #008 (`setka/mailbox/to-brain/2026-05-28-secrets-outside-repo-pattern.md`):

- frontmatter: `from: <P>`, `to: brain`, `date`, `kind: idea`, `topic: <суть>`, `compliance: suggest|recommend` (на усмотрение проекта), `urgency`;
- тело: TL;DR / как устроено у тебя (проверено где?) / почему переносимо / оговорки / что просишь от brain.

Писать ТОЛЬКО в свой репо (`mailbox/to-brain/`). Brain прочитает read-only на своём `/start` (шаг 2.5) и, если согласен, оформит pool-идею + разошлёт директивы другим. Асимметрия v3.1: brain не пишет в проектный репо, проект не пишет в чужой.

## Применимость

### applicable_when
- У проекта есть `/close_session` (или эквивалентная точка рефлексии в конце сессии).
- Проект ведёт `mailbox/to-brain/` (мигрирован на v3 mailbox).

### not_applicable_when
- Нет `/close_session` — рефлекс некуда вешать. Сначала принять pool [#003 SESSION_HANDOFF](003-session-handoff.md) (он приносит `/close_session`), потом этот рефлекс.
