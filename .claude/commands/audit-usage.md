Запускает `.claude/scripts/audit-usage.js` (читает `~/.claude/projects/*.jsonl` через `ccusage` + per-project маппинг), доставляет markdown-отчёт в чат, добавляет содержательную интерпретацию (аномалии, тренды, рекомендации).

## Когда использовать

- **Раз в неделю** в meta-сессии brain_matrica — рутинный аудит (ADR-0003 принцип #1: видимость прежде оптимизации)
- **После крупной сессии** в конкретном проекте — чтобы зафиксировать «во сколько обошлось», для базы сравнения до/после изменений
- **Перед квартальным review** ([token-economy-program.md](../../docs/plans/token-economy-program.md) этап 4)
- **При подозрении на спайк** — счёт за месяц «много», непонятно за что

## Когда НЕ использовать

- ❌ В обычных code-сессиях проектов — там используется `/cost` (built-in, текущая сессия)
- ❌ Для real-time мониторинга — данные обновляются по факту завершения jsonl-событий, не realtime
- ❌ Как замена `git log` или SESSION_HANDOFF — это **только про деньги/токены**, не про что сделано

## Аргументы

- **Без аргументов** → `since = 7 дней назад`
- `/audit-usage 30d` → последние 30 дней
- `/audit-usage 2026-05-01` → начиная с конкретной даты
- `/audit-usage month` → с 1-го числа текущего месяца

Реализация: skill парсит первый аргумент, формирует `--since YYYY-MM-DD`, передаёт в скрипт.

## Шаги

### 1. Определить `since`

Из первого аргумента команды:

```bash
ARG="${1:-7d}"
case "$ARG" in
  7d)    SINCE=$(date -d "7 days ago" +%Y-%m-%d) ;;
  30d)   SINCE=$(date -d "30 days ago" +%Y-%m-%d) ;;
  month) SINCE=$(date +%Y-%m-01) ;;
  *)     SINCE="$ARG" ;;  # assume YYYY-MM-DD
esac
```

(На Windows `date -d` работает через Git Bash. Если не сработает — пользователь передаёт явную дату.)

### 2. Запустить скрипт

```bash
node .claude/scripts/audit-usage.js "$SINCE"
```

Скрипт:
- Запускает `npx -y ccusage@latest session --json --since YYYYMMDD` и `npx ccusage claude --json --since YYYYMMDD`
- Сканирует `~/.claude/projects/*/` для маппинга UUID → проект (с учётом worktrees и archived)
- Выводит markdown с тремя секциями: По проектам / По дням / Top-5 сессий

Первый запуск качает `ccusage@latest` (медленно, ~30-60 сек). Последующие быстрее.

### 3. Воспроизвести вывод скрипта в чат **as-is**

Это уже готовый markdown. Не редактируй, не сокращай, не пересказывай — пользователь хочет точные числа.

### 4. Добавить секцию «🔎 Интерпретация»

Это **то что Claude добавляет от себя**, не скрипт. Анализ:

- **Топ-проект** — какой и почему (контекст активности из памяти / SESSION_HANDOFF)
- **Аномалии** — день/сессия ≥ 2× медианы, неожиданная модель, неожиданный проект (archived с активностью)
- **Cache rate** — норма 80-95%; <70% → что-то странное (новые сессии без overlap)
- **Соотношение моделей** — много Haiku в день обычно = фоновая активность (loops/subagents); мало Opus = хорошо для бюджета, плохо если задачи были сложные
- **Если в нитке есть план token economy** — сравни с предыдущим запуском (если он есть в `git log -- ...`)

Формат: 3-5 буллетов, кратко. Без воды.

### 5. Если есть рекомендация на основе данных — добавить «📋 Action items»

Только если данные **явно** указывают на проблему:

- «Проект X тратит 60%+ при доле задач 20% → провести аудит длинных сессий»
- «Сессия UUID `abc12345` стоила $113 — посмотреть в `~/.claude/projects/<dir>/abc12345.jsonl` что там было»
- «Daily Haiku spike 17/19 мая → проверить не был ли запущен `/loop` или scheduled task»

Если данных мало или всё штатно — этой секции **нет**. Не дописывай ради красоты.

## Что НЕ делает

- ❌ **Не пишет** `docs/AI_USAGE_DASHBOARD.md` или подобный файл (нарушит ADR-0003 принцип #3 — куратируемое, не накопительное). История восстанавливается из `git log -- docs/SESSION_HANDOFF.md` если в close_session фиксировались числа.
- ❌ **Не делает** commit/push результатов. Это аналитическая команда, не оформление.
- ❌ **Не запускает** ничего в проектах. Только локальный read из `~/.claude/projects/`.
- ❌ **Не показывает** не-Claude агентов (Qwen/Codex/etc) — фокус на Claude per ADR-0003.
- ❌ **Не предлагает** автоматических действий (downgrade модели, лимиты) — нарушит ADR-0003 принцип #2.
